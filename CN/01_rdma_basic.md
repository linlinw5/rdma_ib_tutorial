# 第一章：RDMA 基础理论

本章的目的是罗列在了解 RDMA 之前必备的背景知识，假定读者已经具备基本的网络知识与一定的软件开发基础。如果在阅读中有任何感到困惑的地方，建议与AI聊天，把这些背景知识搞透。

## 1.1. 两个基础概念：内核态与用户态

现代操作系统把 CPU 的运行状态分成两层。

**内核态**是操作系统内核运行的模式，拥有最高权限，可以直接操作硬件、管理内存、调度进程。

**用户态**是普通应用程序运行的模式，权限受限，不能直接碰硬件，必须通过"系统调用"请求内核代劳。

这个设计是为了安全和稳定：应用程序出错不会直接把系统搞崩。但代价是：每次系统调用，CPU 都要从用户态切换到内核态，完成操作后再切换回来，这个来回本身是有开销的。

---

## 1.2. 一个普通的 Socket 收发包要经历什么

以一次 TCP 发包为例，看看数据是怎么从你的应用程序到达网卡的。

你在应用程序里调用 write()，这是一个系统调用。CPU 从用户态切换进内核态，内核里的 Socket 层把数据从你的用户态内存缓冲区拷贝一份到内核的发送缓冲区，然后交给 TCP/IP 协议栈做封装，最终提交给网卡。

网卡拿到任务后，并不需要 CPU 帮它搬数据，它通过一种叫做 DMA（Direct Memory Access，直接内存访问）的硬件机制，自己直接读取内核缓冲区里的数据，完成发送。DMA 是现代网卡的基本能力：外设直接与内存之间传输数据，全程不占用 CPU。

收包方向反过来：网卡收到数据，同样通过 DMA 把数据写入内核缓冲区，然后触发中断通知 CPU。内核协议栈处理、解包，再把数据从内核缓冲区拷贝一次到用户态缓冲区，应用程序调用 read() 才能读到。

整个过程，每个包经历至少两次内核与用户态的切换，以及两次内存拷贝。网卡与内核缓冲区之间的搬运由 DMA 完成，不烦 CPU；但内核缓冲区到用户态内存之间的那次拷贝，依然要 CPU 亲自来做。在普通业务场景下，这点开销完全感觉不到。但当你面对的是每秒数百万包的高速网络时，这个开销会大量消耗 CPU，成为性能的瓶颈。

---

## 1.3. DPDK：把内核从数据路径上踢开

既然问题出在内核介入，一个直接的思路是：让应用程序绕过内核，直接跟网卡打交道。这正是 DPDK（Data Plane Development Kit）的核心思想。

DPDK 是一套运行在用户态的网卡驱动框架。它把网卡的控制权从内核驱动手里接管过来，让用户态的程序可以直接读写网卡的收发队列，不经过内核，没有系统调用，没有中断，也不需要拷贝数据进内核再拷出来。

DPDK 的效果很显著：在单机上，它把网络包处理的延迟和 CPU 开销压缩到了极低的水平，是电信、高频交易等对网络性能极度敏感的场景的标配。

但 DPDK 只解决了**单机上内核介入**的问题。数据依然在这台机器的内存里流转，CPU 依然是实际执行收发动作的主体，只是不用再绕内核那一圈了。

---

## 1.4. 并行计算的困境：网络成了新瓶颈

现在把场景切换到另一个维度：大规模并行计算。

深度学习训练用的是几十甚至几百张 GPU，它们分布在多台机器上。训练过程中，各个节点需要频繁地交换梯度数据：—每隔一段时间，所有节点把自己算出的梯度汇总、同步，然后继续下一轮计算。这个通信模式叫做 AllReduce。

问题来了：如果节点之间用普通的 Socket 通信，会发生什么？

发送方的 CPU 需要把 GPU 显存里的数据先搬到 CPU 内存，调用 `write()` 陷入内核，经过协议栈，发出去；接收方的 CPU 被中断打醒，数据从网卡搬进内核缓冲区，再拷贝到用户态内存，再搬回 GPU 显存。整个链路，CPU 在两端都深度参与。

这带来一个根本性的矛盾：GPU 的价值在于并行计算，但通信阶段 CPU 却成了制约速度的环节。GPU 算完等着同步，CPU 在忙着搬数据，算力大量闲置。网络通信的效率，直接决定了整个集群能跑多快。

---

## 1.5. RDMA：让网卡直接读写远端内存

解决这个问题需要一种新的思路：**通信这件事，能不能不经过 CPU？**

RDMA，Remote Direct Memory Access，远程直接内存访问，就是这个思路的实现。

它的核心想法是：绕过操作系统内核，让一台机器的网卡直接把数据写入另一台机器的内存，整个过程 CPU 不介入数据搬运，只需要在发起和完成时做极少量的工作。

具体来说，应用程序事先向系统注册一块内存区域，告诉网卡"这块内存你可以直接操作"。发送方的网卡通过 DMA 从本地内存读取数据，经由网络发出；接收方的网卡通过 DMA 把数据直接写入对端提前注册好的内存区域。两端的 CPU 在这个过程中基本是旁观者。

对并行计算场景来说，这意味着：AllReduce 的通信阶段可以由网卡独立完成，CPU 和 GPU 得以解放出来继续做计算，通信与计算可以真正重叠（overlap）起来。这才让大规模集群的算力得以被充分利用。

这就是 RDMA 存在的理由。

---

## 1.6. RDMA 通信模型及基本概念

理解了 RDMA 的动机之后，在真正使用它之前还需要建立一套概念框架。RDMA 的编程模型和传统 Socket 差异很大，有一套自己的术语体系，但逻辑并不复杂。

---

### 1.6.1 传输类型：RC、UC 与 UD

RDMA 支持三种传输类型，区别在于可靠性保证和连接方式。

**RC（Reliable Connection，可靠连接）** 是使用最广泛的类型。它在两个端点之间建立一条连接，保证数据按序、无丢失地到达，类似 TCP。绝大多数 RDMA 应用（包括 AI 训练集群的通信库）都跑在 RC 上。

**UC（Unreliable Connection，不可靠连接）** 同样是点对点连接，但不提供重传保证，数据可能丢失。实际场景中很少使用。

**UD（Unreliable Datagram，不可靠数据报）** 不建立连接，一个端点可以向任意对端发送数据，类似 UDP。没有可靠性保证，但灵活，适合广播、组播等场景。

#### 建立连接时的区别

**RC 和 UC** 都需要提前建立 QP 连接，双方交换：

```
QPN（Queue Pair Number）
PSN（Packet Sequence Number）
GID / LID
```

**UD** 不建连接，发送时临时指定目标，通过 **AH（Address Handle）** 描述对端地址。

#### 发包时的区别

**RC（Reliable Connection）**

```
发送方:  Post Send
         ↓
         硬件自动分片（超过 MTU 则拆包）
         硬件维护 PSN，自动重传丢包
         硬件等待 ACK
接收方:  收到后硬件自动发 ACK
         应用层看到的是完整数据，无需关心分片
```

- 支持 RDMA Write / RDMA Read / Send / Atomic
- 每个连接独占一对 QP，**QP 数量 = 连接数**，大规模场景 QP 爆炸是痛点

**UC（Unreliable Connection）**

```
发送方:  Post Send
         ↓
         硬件分片发送
         没有 ACK，没有重传
         丢包就丢了，对方不知道
接收方:  收到哪些算哪些
```

- 只支持 RDMA Write / Send，不支持 RDMA Read
- 实际几乎没人用，介于 RC 和 UD 之间，两头的优点都不够极致

**UD（Unreliable Datagram）**

```
发送方:  Post Send + 指定 AH（目标地址）
         ↓
         单包发送，不能超过 MTU（4096B）
         不分片，不重传，不 ACK
接收方:  每个 Recv buffer 必须比实际数据大 40 字节
         （头部有 GRH，Global Routing Header）
```

- 只支持 Send/Recv，**不支持任何 RDMA 单边操作**
- 一个 QP 可以和任意多个对端通信
- MPI 的 barrier、allreduce 等集合通信早期实现常用 UD

#### 可靠性机制在哪层实现

```
RC:  可靠性在 RDMA 硬件/驱动层
         丢包重传、乱序重组、流控 —— 应用完全不感知

UC:  没有可靠性，应用自己处理或直接忽略

UD:  没有可靠性，且单包不超 MTU，应用自己分片/重传
         （NCCL 的 IB UD 模式就自己实现了一套可靠层）
```

对比 TCP/IP：

```
TCP  ≈ RC    内核处理可靠性
UDP  ≈ UD    应用自己处理（或不处理）
```

#### 实际使用场景

| 场景                      | 用哪种                |
| ------------------------- | --------------------- |
| NCCL 点对点通信（GPU 间） | RC（主流）            |
| MPI 集合通信              | RC 或 UD              |
| 存储协议（NVMe-oF、iSER） | RC                    |
| RoCEv2 大规模 AI 集群     | RC + DCB/PFC 保障无损 |
| 需要组播 / 一对多         | UD（RC/UC 做不到）    |
| 低连接数、追求极致延迟    | RC                    |
| 连接数极多（万级节点）    | UD（避免 QP 爆炸）    |

### 1.6.2 Verbs：RDMA 的编程接口

RDMA 的编程接口统一叫做 **Verbs**。你可以把它理解成 RDMA 世界里的"Socket API"，不管底层是 InfiniBand 还是 RoCE，上层应用都通过 Verbs 来操作。Verbs 围绕几个核心对象展开。

**内存注册（Memory Registration，MR）**

前面提到，RDMA 要求网卡能直接访问应用程序的内存。但网卡不能随便访问任意一块内存，应用程序必须先向系统"注册"一块内存区域，让操作系统把这块内存锁定（不允许被换出到磁盘），并告诉网卡它的物理地址。注册成功后，这块内存就可以作为 RDMA 操作的来源或目标。注册是有代价的，所以实际使用中通常提前注册好一个大的内存池，反复复用。

**Queue Pair（QP，队列对）**

RDMA 的收发不通过内核的 Socket，而是通过 QP 来完成。一个 QP 包含两条队列：发送队列（Send Queue，SQ）和接收队列（Receive Queue，RQ）。你想发数据，就往 SQ 里提交一个描述符；对端想接收数据，就提前往 RQ 里放好接收缓冲区。RC 模式下，两端各有一个 QP，通信前需要建立连接，把两个 QP 配对绑定。

**Work Queue Element（WQE，工作队列元素）**

提交到队列里的每一条"任务描述"就是一个 WQE。它告诉网卡：做什么操作、数据在哪块内存、长度是多少、发到哪里。CPU 把 WQE 写进队列，然后敲一下网卡的"门铃"寄存器，网卡就开始自己干活，不再需要 CPU 参与。

**Completion Queue（CQ）与 CQE**

网卡完成一个 WQE 之后，需要通知应用程序。它的做法是往 **CQ（完成队列）** 里写入一条 **CQE（完成队列元素）**，记录这次操作的结果——成功还是失败、传输了多少字节。应用程序轮询或等待 CQ，拿到 CQE 才知道某次操作已经结束。

**WR 与 WC：用户视角的命名**

在用户态编程接口里，WQE 对应的结构体叫做 **WR（Work Request，工作请求）**，CQE 对应的结构体叫做 **WC（Work Completion，工作完成）**。名字不同，指的是同一套机制在用户层的体现：你提交 WR，事后收到 WC。

### 1.6.3 双边通信与单边通信

RDMA 的操作分成两类，核心区别在于对端 CPU 是否需要参与。

**双边通信（Two-sided）** 指的是两端都需要主动配合。发送方提交一个 Send 操作，接收方必须提前在 RQ 里准备好接收缓冲区，数据才能落地。两端的 CPU 都要介入——至少要负责提前准备好接收侧的 WQE。

**单边通信（One-sided）** 是 RDMA 最独特的能力。发送方可以直接读写对端的内存，而对端 CPU 完全不知情、不参与。这要求对端提前注册好内存并把访问凭证（内存的地址和权限密钥）告知发送方，之后发送方就可以绕过对端 CPU，直接让网卡操作那块内存。

---

### 1.6.4 四种基本操作

RDMA 的四种操作，本质上是两个维度的组合：**谁发起**（发起方 vs 对端 CPU）× **谁决定数据落在哪里**（本地 vs 远端）。理解这两个维度，后面的连接建立过程——为什么双方要交换 rkey、GID、内存地址——就会顺理成章。

**Send（双边操作）**

最基础的操作，最接近传统 Socket 的 `send()`。发送方把数据推向对端，但数据最终落在哪块内存，由**接收方**决定。接收方必须提前在自己的 Receive Queue 上挂好 WQE，指定好接收缓冲区；发送方对此一无所知，它只管发。

这就是"双边"的含义：**双方的 CPU 都要参与**：发送方 CPU 提交 Send WQE，接收方 CPU 提前准备 Recv WQE。数据路径上不需要 CPU 搬运，但控制路径上双方都要行动。

Send 的典型用途是传递元数据、控制消息，或者用于建连阶段：比如两端互相告知自己的内存地址和 rkey，以便后续做单边操作。

**RDMA Write（单边操作）**

发送方把数据直接写入**对端指定的内存地址**。对端 CPU 完全不参与，它不需要提前挂 Recv WQE，数据就悄悄出现在那块内存里。

但这里有个关键问题：发送方怎么知道该写到对端内存的哪个地址？

答案是：**对端需要提前告诉发起方**。对端通过带外通道（通常就是一次 Send 操作）把三样东西发给发起方：目标虚拟地址、这块内存区域的长度，以及一个叫 **rkey**（Remote Key）的令牌。

rkey 是这里的核心概念。RDMA 的单边操作绕过了对端 CPU，但不能绕过对端的安全检查，否则任何节点都可以随意读写远端内存。rkey 就是 RDMA 硬件执行访问授权的凭证：发起方在 Write 请求里携带 rkey，对端网卡（RNIC）收到请求后，会查自己的内存保护表，验证这个 rkey 是否与目标地址对应、是否有写权限、是否已过期。只有验证通过，DMA 写入才会真正发生。

可以把 rkey 理解成一把临时授权的"保险箱钥匙"：对端把钥匙交给你，你才能打开特定的那个格子，而且钥匙随时可以吊销（注销 MR）。

RDMA Write 非常适合高吞吐的数据推送场景：存储系统写数据块、AI 训练中参数服务器分发梯度，都是典型用例。

**RDMA Read（单边操作）**

发起方主动从**对端内存**里拉取数据，搬回本地。同样，对端 CPU 不参与，对端 RNIC 直接从内存读出数据返回。

RDMA Read 同样需要 rkey，原因与 Write 完全对称：发起方需要告诉对端网卡"我要读的是这个地址、这段长度，这是你给我的 rkey"，对端网卡验证权限后才执行读取。

与 Write 相比，Read 多了一个网络往返：发起方发出 Read 请求，对端网卡把数据打包发回，发起方收到后才算完成。

RDMA Read 适合按需拉取远端数据的场景：比如分布式键值存储的 GET 操作，或者计算节点从参数服务器按需拉取某一层权重。

**Atomic（原子操作）**

RDMA Atomic 是一种**单边操作**，和 Write/Read 一样不需要对端 CPU 参与，但特殊之处在于它是**原子的读-改-写**，硬件保证操作不可分割。

在对端内存上执行**不可分割的读-改-写**操作。有两种：

- **Fetch-and-Add**：读出当前值，加上一个数，写回，返回旧值。
- **Compare-and-Swap**：读出当前值，若等于预期值则替换为新值，返回旧值。

"不可分割"是关键。打个比方：你尝了汤觉得淡，去隔壁拿盐；与此同时室友也尝了，也去找盐。你回来加了刚好，室友回来不知情又加一次，汤就齁咸了。问题在于"尝"和"加"是两步，中间有空隙可以被插入。原子操作把这两步绑成一个硬件级别的原语——在这两个动作之间，没有任何其他操作能插入。

在多节点并发访问同一块远端内存时，这个保证至关重要。分布式锁、序列号生成器、引用计数，这些需要"只有一个赢家"的场景，都可以用 Atomic 来实现。

例如：RDMA Write 可以直接写对端内存，但如果多个节点同时写同一个地址就会有竞争：

```
节点A读到值=5，节点B读到值=5
节点A写入6，节点B也写入6   ← 实际应该是7
```

Atomic 操作在硬件层面保证读-改-写是不可分割的，解决这个问题。GPU 集群里 NCCL 的 AllReduce 同步原语就依赖这个机制。

**限制**

- 只有 RC QP 支持，UC/UD 不支持
- 操作数据宽度固定 8 字节（64位），不能更大
- 目标地址必须 8 字节对齐

Atomic 同样需要 rkey，且对内存对齐有严格要求（通常要求 8 字节对齐），因为底层依赖的是 PCIe 总线和内存控制器的原子访问能力。

---

要让这四种操作真正跑起来，双方在正式传输数据之前必须先完成一次"参数交换"，主要包含以下内容：

- **QPN（Queue Pair Number）**：每个 QP 在本地 RNIC 上有一个编号，相当于端口号。发出去的报文要写明"送到对端的哪个 QP"，对端 RNIC 收到后才知道把这个请求交给哪个 QP 处理。没有 QPN，报文到了对端就不知道往哪投递。
- **PSN（Packet Sequence Number）**：初始序列号。RC 传输模式下，双方用 PSN 来检测丢包、乱序和重复报文。连接建立时各自随机生成一个初始 PSN 告知对端，后续每发一个报文 PSN 递增，对端据此判断报文是否完整有序。
- **虚拟内存地址（Virtual Address）**：单边操作（Write / Read / Atomic）的目标。发起方需要知道"写到对端内存的哪个地址"或"从对端内存的哪个地址读"。这个地址是对端注册 MR 时的虚拟地址，需要对端主动告知。
- **rkey（Remote Key）**：访问授权令牌，与内存地址配套使用。二者缺一不可：地址说明"在哪里"，rkey 说明"有没有资格"。

把四种操作放在一起，可以看出一个规律：

| 操作       | 对端 CPU 参与 | 需要 rkey | 需要对端提前挂 Recv WQE |
| ---------- | ------------- | --------- | ----------------------- |
| Send       | 是            | 否        | 是                      |
| RDMA Write | 否            | 是        | 否                      |
| RDMA Read  | 否            | 是        | 否                      |
| Atomic     | 否            | 是        | 否                      |
