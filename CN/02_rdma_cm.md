# 第二章：RDMA 连接管理

RDMA 在传输时需要填充包头。以 Write 操作为例，发送方的网卡需要知道对端的 QPN、目标内存地址、rkey 等关键信息，才能构造出合法的报文发出去。这些信息在通信开始之前必须由双方提前协商好。

对于 bootstrap 阶段，业界普遍有两种解法：

**带外通信（Out-of-band）**：用另一套传输机制先把这些信息交换好，再开始 RDMA 通信。最常见的做法是用 TCP Socket——两端先建一条 TCP 连接，把各自的 QPN、内存地址、rkey 等信息互相发过去，握手完成后再切换到 RDMA 传输。

**带内通信（In-band）**：直接用 RDMA 网络自身提供连接建立和地址交换的服务，不依赖额外的传输通道。这套机制叫做 **RDMA CM**（Connection Manager）。

---

## 2.1 需要交换哪些信息

在建立连接之前，双方至少需要交换以下信息，才能让 QP 进入可以收发数据的状态。

**QPN（Queue Pair Number）**：对端 QP 的编号，相当于 RDMA 世界里的"端口号"，告诉网卡数据要交给对方哪个 QP。

**PSN（Packet Sequence Number）**：初始报文序列号，用于接收端校验报文顺序。RC 和 UC 模式需要交换 PSN；UD 模式不建立连接，不需要。

**虚拟内存地址与 rkey**：这两项只有单边操作（例如：Write、Read）才需要。发送方要直接操作对端内存，必须知道目标内存的虚拟地址以及对应的 rkey（远端访问密钥）。双边操作（Send）由接收方自己决定数据落到哪块缓冲区，发送方不需要知道对端的内存地址，因此也不交换这两项。

**GID/LID（IB 的寻址信息）**：有个反直觉的东西，既然双方在交换通信信息，那么肯定是已经知道对端的地址了，为什么这里还要交换 IB 寻址信息呢？这其实是个工程实践的问题，应用软件（例如：AI训练框架）往往活在 IP 的世界里，但是 GPU/CPU 需要通过 IB 访问远端机器的内存，这就需要知道对端在 IB 中的寻址信息。

---

## 2.2 带外方式：用 TCP 交换信息

带外方式的思路简单直接：在 RDMA 通信之前，先用 TCP 建一条普通的连接，把上述信息互相发过去，两端各自据此完成 QP 的状态配置，再开始 RDMA 传输。

**TCP 连接的生命周期是一个实现选择，协议层面没有强制要求。** 信息交换完成后，QP 本身已经维护了连接状态，那条 TCP 连接在原理上可以立即断开。但很多实现会选择保留它，当作心跳通道或异常检测手段。因为一旦 TCP 断开，就可以判断对端出了问题，触发重连或错误处理。保留还是断开，取决于开发者对可靠性和资源消耗的权衡，没有唯一正确的答案。

**底层网络类型会影响带外方式的部署架构** 当底层是 InfiniBand 时，由于 IB 网络本身不承载 TCP/IP 流量，因此带外通信必须依赖一套独立的以太网。当底层是 RoCEv2 时就简单得多：RoCEv2 本身就跑在以太网上，带外的 TCP 连接直接复用现有的以太网链路，不需要任何额外硬件。这个差异很直观：IB 方案要为"控制面"单独铺一张网，RoCEv2 方案则控制面和数据面共用一张以太网。

---

## 2.3 带内方式：RDMA CM

带外方式的问题在于，它把连接管理的责任全扔给了应用程序：你得自己写 TCP 握手逻辑，自己定义地址交换的格式，自己决定连接的生命周期。对于需要管理大量连接的系统，这套代码并不轻松。

RDMA CM（Connection Manager）是 Linux 内核提供的连接管理机制，它把上面那套握手过程封装起来，通过 RDMA 网络自身完成地址解析和连接协商，不需要应用程序自己维护一条额外的 TCP 通道。

它最实用的地方在于，提供了一套与 Socket API 高度对应的编程接口。换句话说，**RDMA CM 就是用来平替带外 TCP 那套握手代码的**，熟悉 TCP 编程的开发者几乎可以照着 Socket 的思路写下来。两套 API 的对应关系大致如下：

| Socket API  | RDMA CM API         | 作用             |
| ----------- | ------------------- | ---------------- |
| `socket()`  | `rdma_create_id()`  | 创建一个通信端点 |
| `bind()`    | `rdma_bind_addr()`  | 绑定本地地址     |
| `listen()`  | `rdma_listen()`     | 监听连接请求     |
| `connect()` | `rdma_connect()`    | 发起连接         |
| `accept()`  | `rdma_accept()`     | 接受连接         |
| `close()`   | `rdma_destroy_id()` | 销毁端点         |

在 `rdma_connect()` / `rdma_accept()` 的过程中，RDMA CM 会自动完成 QPN、PSN 等信息的交换和 QP 的状态迁移，开发者不必手动处理这些细节。连接建立之后，数据传输阶段仍然走标准的 Verbs 接口，CM 只负责握手这一段，不介入数据路径。

最关键的是，通过 RDMA CM 建立连接完全是通过带内方式来完成的，这意味着对于 IB 网络来说，无需额外的以太网支持便可完成连接的建立。

---

## 2.4 IB 寻址信息

在 2.1 中我们提到，通信双方要交换 IB 寻址信息，结合底层传输网络（IB/RoCEv2）的不同以及解法的（带内/带外）不同，大致总结如下：

RoCEv2 场景相对简单。RoCEv2 跑在以太网上，GID 直接从网卡的 IP 地址派生出来，两者是同一个地址的两个视角。

- 带外 TCP 方式下，双方把各自的 GID 打包进握手消息互相发送；
- RDMA CM 方式下，应用只需要给出对端 IP 地址，CM 内部可以解析出 GID，应用层完全不需要感知 GID 的存在。

备注：GID 在 RoCEv2 里身兼两职：对 RDMA 层来说它是端点标识，对 RNIC 硬件来说它同时也是构建外层 IP 头的数据来源。

纯 IB 场景则更复杂，因为 IB 有自己独立的寻址体系，LID 由 SM 分配，GID 也与 IP 地址没有任何关系。

- 带外 TCP 方式下，双方必须把 LID 或 GID 通过 TCP 通道显式交换：ibv_modify_qp 把 QP 推进到 RTR 状态时，ah_attr 里必须填入对端的 LID/GID，这个信息网卡自己查不到，只能靠对端发过来。
- RDMA CM 方式则由 CM 内部处理：传统做法是借助 IPoIB 把 IP 地址映射到 GID，再去查 SA Path Record；也可以直接用原生 IB 地址（GID）跳过映射步骤，拿着 GID 直接查 SA Path Record。地址和路径解析完成后，CM 在 IB 网络内部完成后续的握手，整个过程对应用透明。

备注：关于 GID 和 IB 寻址的细节，这里不必过于纠结，在第八章 infiniband 基础部分会有详细的介绍。

## 2.5 小结

两种方式解决的是同一个问题：在 RDMA 数据传输开始之前，把 QPN、PSN、内存地址、rkey 这些参数安全地送到对端。

从实际使用来看，不同软件栈的选择差异很大：

IB 场景下，两种方式并存，分界线在于软件栈的定位。

- 生产环境的 IB 集群几乎都配有管理以太网，带外方式可以直接复用它。NCCL（NVIDIA Collective Communications Library，见下一个小节的介绍） 就是这样做的：bootstrap 阶段走管理网上的 TCP socket 交换 QPN 和 rkey，数据面走 IB Verbs，全程不依赖 RDMA CM。选择 TCP bootstrap 有一个更根本的原因：框架完全活在 IP 地址的世界里，不需要感知 IB 的寻址体系（LID、GID）。IB 网络被当成一条纯粹的高速通道，rank 的发现和连接管理全部在 IP 层完成，bootstrap 代码在 IB 和 RoCEv2 上可以完全复用，多 QP、多 rail 的拓扑管理和容错逻辑也由框架自己掌控。
- HPC 领域的 MPI/UCX 则走另一条路，用 IB 自身的 UD（Unreliable Datagram）做 wireup，在 IB 网络内部完成 RC QP 的连接建立，真正做到不依赖任何带外通道。
- 存储协议栈（NVMe-oF、iSER）则更倾向于 RDMA CM，看重的是标准化接口和运维可操作性。

RoCEv2 场景下，AI 训练框架的逻辑与 IB 场景完全一致。

- NCCL、PyTorch distributed 同样用 TCP socket 或 MPI 集合操作完成 bootstrap，RDMA 连接建立完成后带外通道退出数据路径。
- 存储协议栈同样偏向 RDMA CM。底层换成 RoCEv2，软件栈的选择逻辑没有任何变化。

这背后有一条贯穿始终的分界线：应用框架选带外，协议栈选带内。 框架需要自己掌控寻址、拓扑、容错，带外方式给了它完整的控制权；协议栈需要标准化接口和互操作性，RDMA CM 的 Socket 类 API 恰好满足这个需求。

无论哪种方式握手结束之后，数据路径是完全一样的：标准的 Verbs 操作，相同结构的 IB 报文。后面的章节会通过抓包，让你看到这些在 bootstrap 阶段交换的参数是怎么出现在实际报文里的。

## 2.6 补充知识点：NCCL 与 RDMA 的关系

NCCL（NVIDIA Collective Communications Library）是 NVIDIA 推出的集合通信库，专为多 GPU 分布式训练场景设计。在大模型训练中，数以百计乃至千计的 GPU 需要在每一轮迭代后同步梯度，这个过程涉及复杂的跨节点数据交换。NCCL 将这一复杂性封装起来，向上层 AI 框架（PyTorch、JAX 等）提供 AllReduce、AllGather、Broadcast 等集合通信原语，开发者只需调用一个 `ncclAllReduce()`，跨节点的梯度同步便自动完成，无需关心任何网络细节。

NCCL 与 RDMA 的关系，可以用 **C 语言与汇编**来类比。RDMA 是"汇编"：操作精确但底层，使用者需要亲自管理 QP、MR、WR、CQ，明确告知硬件"将数据写到对端的某个内存地址"。NCCL 则是"C 语言"：它在 RDMA 语义之上构建了更高层的抽象，将 QP 建立、内存注册、ring 算法分片、多步传输等细节全部封装在内部，对上层框架完全透明。就像 C 程序员不需要知道 `c = a + b` 被编译成了多少条汇编指令，AI 框架工程师也无需知道一次 AllReduce 背后究竟发出了多少个 RDMA WRITE。

NCCL 让 AI 应用开发者从网络复杂性中解放出来，但从网络工程师的角度，它却更像一个"黑盒"。因为 NCCL 的每一个集合通信操作，在底层最终都会被分解为一系列 RDMA 动作，然后在 IB 或 RoCE 网络上逐一执行。所以，对于网络工程师来说，理解 AI Networking，依然要从 RDMA 入手。
