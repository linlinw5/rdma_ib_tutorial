# 第八章：InfiniBand 基础理论

读到这里，我们已经了解了RDMA 的基本网络行为，接下来的我们一起快速浏览一下 RDMA 的原生传输通道：InfiniBand。

## 8.1 为什么要专门学 InfiniBand

RDMA 这套语义（Queue Pair、Verbs、单边操作、内存注册）最初就是为 InfiniBand 设计的。RoCE 做的事情，本质是"把 IB 的传输层语义搬到以太网上跑"。也就是说，前面七章所涉及的所有 RDMA 概念，源头都在 IB。RoCE 是移植，IB 是原版。

理解了 IB 是怎么设计的，我们才能回答几个 RoCE 世界里说不清楚的问题：为什么 RoCE 要费那么大劲去搞无损网络（PFC、ECN、DCQCN）？为什么 IB 不需要这些却天生无损？为什么 IB 的延迟能稳定做到微秒级而 RoCE 总要调优？这些问题的答案，都藏在 IB 的设计哲学里。看懂了原版，才能看清移植版的妥协在哪里、未来会往哪走。

这一章重点是两件事：

- 理解 IB 区别于以太网的设计思想；
- 把 IB 资料里那一堆让人头大的专用名词扫盲清楚。

---

## 8.2 InfiniBand 是什么

InfiniBand 是一套开放标准的互联协议，由 InfiniBand 行业协会（IBTA）制定，成员包括 NVIDIA、Intel、IBM、HPE、Oracle 等。它在 2002 年前后起步，目标从一开始就很明确：为高性能计算和数据中心提供高吞吐、低延迟、可扩展的服务器互联。今天它是 AI 训练集群和 HPC 的主流互联方案。

但这些描述对理解它的本质帮助不大。真正的关键是一句话：

**InfiniBand 是一套从物理层到传输层垂直整合、专为 RDMA 而生的网络架构。**

我们熟知的以太网 + TCP/IP 是分层演进、各自独立的。以太网负责二层，IP 负责三层，TCP 负责传输，每一层由不同的标准在不同年代拼起来，彼此之间没有统一设计。InfiniBand 不一样，它从底层链路到上层传输是一整套统一规范，所有层都围绕"让数据以最小的 CPU 介入、最低的延迟跨网络搬运"这一个目标协同设计。这种垂直整合，正是它和以太网世界一切差异的根源。

---

## 8.3 InfiniBand 与以太网的根本不同

把 IB 和我们熟悉的以太网/IP 世界放在一起对比，几个最关键的设计差异会立刻浮现出来。

**第一，无损 vs 尽力而为。**

以太网的设计哲学是"尽力而为"（best-effort）：交换机缓冲区满了就直接丢包，靠上层的 TCP 去发现丢包、重传、降速。丢包是常态，可靠性是上层补出来的。

InfiniBand 的链路层天生无损。它用一种叫**基于信用的流量控制**（credit-based flow control）的机制：接收端会明确告诉发送端"我还有多少缓冲区可以收"，发送端只在对方有空间时才发数据。缓冲区不够，发送端就停下来等，而不是把包丢掉。所以在正常运行的 IB 网络里，根本不存在因为拥塞而丢包这回事。

这一点直接解释了 RoCE 的核心痛点：RoCE 要在以太网上跑 IB 的传输语义，但以太网会丢包。于是 RoCE 不得不引入 PFC、ECN 这些机制，费尽力气把以太网"改造"成无损的，而这正是 IB 与生俱来、不需要任何额外配置就有的能力。我们在 RoCE 部署里调过的那些流控参数，本质上都是在补 IB 免费送的那套无损的能力。

**第二，集中管理 vs 分布式自治。**

以太网是去中心化的：每台交换机自己学习 MAC 地址、自己跑生成树、自己做决策，没有一个全局的大脑。这很灵活，但也意味着没人对整张网络有全局视图。

InfiniBand 是集中管理的。每个 IB 子网里有一个**子网管理器**（Subnet Manager，SM），它负责发现整个网络的拓扑、给每个端口分配地址、计算所有的转发路径、再把转发表下发到每台交换机。整张网络由一个中心大脑统一规划。这带来了即插即用（节点接上就被自动发现和配置）和全局最优路由的能力，代价是需要这么一个管理实体存在。

以太网发展到后期也在谈“自动发现、自动部署”，例如 Cisco ACI、DNAC 里面经常能听到这些名词，大多要依赖一组外部的 controller。但是“自动发现、自动部署”却是 IB 与生俱来拥有的能力。

**第三，链路层寻址 vs IP 寻址。**

以太网/IP 世界里，跨网络通信靠的是 IP 地址和逐跳路由。InfiniBand 子网内部不用 IP，它用一种叫 **LID** 的本地标识符在链路层直接转发。

**第四，硬件传输层 vs 软件协议栈。**

以太网世界里，TCP/IP 协议栈跑在内核软件里，靠 CPU 执行。InfiniBand 把整个传输层（分段、重组、确认、重传）做进了网卡硬件。这正是 RDMA 能绕过内核、绕过 CPU 的物理基础，协议处理本来就不在 CPU 上，而在网卡的芯片里。

把这四点连起来看，IB 的形象就清楚了：它是一个由中心大脑统一调度、链路层自带无损保证、寻址和协议处理都下沉到硬件的网络。它不是"更快的以太网"，而是另一套范式。

## 8.4 物理层速率代号扫盲：SDR 到 XDR 到底是多少 G

IB 资料里最劝退新手的，就是 SDR、QDR、HDR、NDR 这堆代号，从名字完全看不出速率，每次都得查表。这一节把它讲透。

先理解一个最关键的概念：**IB 端口的带宽 = 单条 lane 的速率 × lane 的数量。**

IB 用多条物理通道（lane）并行传输来叠加带宽，常见的端口宽度有 1x、4x、12x。**绝大多数服务器网卡和资料里说的速率，默认都是 4x。** 这就是查表对不上的最大根源：同一个代号 NDR，单条 lane 是 100G，但我们平时听到的"NDR 400G"指的是 4x（4 条 lane × 100G）。因此，看资料时一定要分清在说 per-lane 还是 per-port。

下面这张表，per-lane 是规范定义的基准，4x 列是实际最常打交道的端口速率：

| 代号 | 全称               | 单 lane 速率 | 4x 端口速率 | 编码方式 | 起始年代 |
| ---- | ------------------ | ------------ | ----------- | -------- | -------- |
| SDR  | Single Data Rate   | 2.5 Gb/s     | 10 Gb/s     | 8b/10b   | 2001     |
| DDR  | Double Data Rate   | 5 Gb/s       | 20 Gb/s     | 8b/10b   | —        |
| QDR  | Quad Data Rate     | 10 Gb/s      | 40 Gb/s     | 8b/10b   | 2007     |
| FDR  | Fourteen Data Rate | 14.0625 Gb/s | 56 Gb/s     | 64b/66b  | 2011     |
| EDR  | Enhanced Data Rate | 25 Gb/s      | 100 Gb/s    | 64b/66b  | 2014     |
| HDR  | High Data Rate     | 50 Gb/s      | 200 Gb/s    | 64b/66b  | 2018     |
| NDR  | Next Data Rate     | 100 Gb/s     | 400 Gb/s    | 64b/66b  | 2022     |
| XDR  | (next) Data Rate   | 200 Gb/s     | 800 Gb/s    | 64b/66b  | 2023+    |

几个看表时要注意的坑：

**命名毫无规律可循，别想记规则。** SDR/DDR/QDR 早期还能看出"单倍/双倍/四倍"的意思，到 FDR（Fourteen，指 14G）之后就纯粹是市场命名了——F、E、H、N、X 既不是缩写也不是顺序，只能硬记或查表。

**标称速率不等于有效速率，差在编码开销。** 物理线路上传的不只是数据，还有用于时钟同步和纠错的冗余位。SDR/DDR/QDR 三代用的是 8b/10b 编码：每 10 个比特里只有 8 个是有效数据，开销高达 20%。所以 QDR 4x 标称 40G，实际有效吞吐只有约 32G。从 FDR 开始改用 64b/66b 编码，开销骤降到约 3%，效率高得多。看到"理论带宽"和"实际带宽"对不上，就是这个原因。

**XDR 及更高。** XDR 在 2023 年由 IBTA 发布规范，单 lane 200G、4x 端口 800G，目前主要用在最新的 GPU 集群里。IBTA 路线图上还有更高的代际在规划中，速率仍在持续翻倍。

实践中有一个换算心法：**听到一个代号，先默认它是 4x，然后用上表查 4x 那一列**，基本就不会错。需要精确时再回头确认是不是 per-lane 或 12x。

## 8.5 核心概念：地址与标识

IB 里有三个最基础的标识符：GUID、LID、GID，初学者极容易混淆。用一个类比就清楚了：

> GUID 像设备的 MAC 地址（出厂烧死、永不改变），LID 像子网内临时分配的门牌号，GID 像全局标识地址。

**GUID（Globally Unique Identifier，全局唯一标识符）**

每个 IB 设备（网卡、交换机、端口）出厂时由厂商烧录的 64 位永久标识，全球唯一，断电重启也不变。它标识的是"这个硬件是谁"，类似以太网的 MAC 地址。常见的有 Node GUID（标识整个设备）和 Port GUID（标识具体端口）。

**LID（Local Identifier，本地标识符）**

子网管理器在设备接入时**动态分配**的 16 位地址，仅在当前子网内有效。IB 子网内部的链路层转发，靠的就是 LID，交换机看包头里的目的 LID 决定往哪个端口送。因为是 16 位，一个 IB 子网理论上最多约 4.8 万个 LID 地址，这也正是"单子网最多约 48000 节点"这个数字的由来。

**GID（Global Identifier，全局标识符）**

128 位的全局地址，格式和 IPv6 一样，用于**跨子网**通信。当数据需要离开本子网、经过 IB 路由器到达另一个子网时，就靠 GID 来寻址。GID 由两部分组成：子网前缀（subnet prefix，标识哪个子网）+ 接口标识（通常由 Port GUID 派生而来）。

三者的关系可以这样串起来：GUID 是硬件的永久身份，LID 是它在本子网拿到的临时门牌，GID 是它在跨子网时用的全局地址。子网内通信只需要 LID，跨子网才动用 GID。

这里我想特别的说一句，不要用 TCP/IP 世界里那套二三层地址映射（ARP）以及“洋葱式”的封装/解封装的概念来理解 IB，IB 的世界里没有对应的概念。

我们会在第十章介绍 IB 的寻址架构，这里只需要对这些概念有个基础的了解即可。

## 8.6 IB 的管理面：SM、SMA、SA、MAD 这一家子

IB 是集中管理的，核心就是子网管理器。但围绕它有一圈名词：SM、SMA、SA、MAD、SMP、GMP，这些是读 IB 资料时绕不开的，放在一起讲清楚它们的关系。

**SM（Subnet Manager，子网管理器）** 是整个子网的大脑。它做四件事：发现拓扑（网络里有哪些设备、怎么连的）、给每个端口分配 LID、计算转发路径、把转发表下发到每台交换机。一个子网里可以有多个 SM，但只有一个是 master，其余作为 standby 待命，master 挂了就接管——保证管理面不单点故障。

**SMA（Subnet Management Agent，子网管理代理）** 是每个 IB 设备里内置的一个小代理。SM 是大脑，SMA 就是大脑安插在每个节点上的"联络员"。SM 要查询某个设备的状态、或者给它配置 LID，都是通过和这个设备的 SMA 通信来完成的。

**MAD（Management Datagram，管理数据报）** 是管理面通信用的标准消息格式。SM 和 SMA 之间所有的对话（查询、配置、上报）都封装成 MAD 来传递。可以理解为 IB 管理面的"通用信封"。

MAD 根据用途分成两类：

- **SMP（Subnet Management Packet，子网管理包）**：专门用于子网管理的 MAD，比如 SM 发现拓扑、分配 LID、配置端口，走的都是 SMP。SMP 有个特殊能力：它可以在 LID 还没分配好的时候就工作（用一种叫 directed routing 的方式按路径逐跳走），这样 SM 才能在网络刚上电、谁都还没有地址的"鸡生蛋"阶段完成初始发现。
- **GMP（General Management Packet，通用管理包）**：用于子网管理之外的其他管理任务，比如性能监控、设备信息查询等。

**SA（Subnet Administrator，子网管理员）** 是 SM 对外提供的查询服务接口。SM 掌握着全网信息（谁有什么 LID、路径怎么走），普通节点想知道"我要和某个节点通信，该走哪条路、用什么参数"，就向 SA 发查询请求。因此，SA就是一个掌握全网路径信息、供大家查询的"信息台"。

把这一家子串起来：**SM 是大脑，靠遍布各节点的 SMA 当联络员，彼此之间用 MAD 这种信封通信（管理子网用 SMP，其他用 GMP），而 SA 是 SM 开放给所有节点的查询窗口。**

## 8.7 IB 包结构：包头里都有什么

IB 包和以太网包一样，是负载（payload）被各层的头部和尾部"夹"在中间形成的。下面是一个完整 IB 数据包的字段布局：

```
                        recomputed per hop, covers whole packet ──┐
  invariant end-to-end, covers only unchanging fields ─┐          │
                                                       │          │
┌──────────┬──────────┬──────────┬──────────────┬──────────┬──────────┐
│   LRH    │   GRH    │   BTH    │   Payload    │   ICRC   │   VCRC   │
│  Local   │  Global  │   Base   │     Data     │ Invariant│   Link   │
│  Routing │  Routing │ Transport│   payload    │   CRC    │   CRC    │
│  Header  │  Header  │  Header  │              │          │          │
└──────────┴──────────┴──────────┴──────────────┴──────────┴──────────┘
  8 bytes   40 bytes   12 bytes    variable       4 bytes    2 bytes
  always     inter-     always                    always     always
            subnet
             only
     │          │          │
     │          │          └─ dst QP, opcode (Send/Write/Read), PSN
     │          └─ src/dst GID, inter-subnet routing only
     └─ src/dst LID, intra-subnet forwarding

```

逐个字段的作用：

**LRH（Local Routing Header，本地路由头）**：包含源和目的 LID，是子网内链路层转发的依据。交换机就看这个头决定往哪个端口转。每个 IB 包都有。

**GRH（Global Routing Header，全局路由头）**：包含源和目的 GID，只有在需要跨子网路由时才出现。同子网内通信时这个头是省略的。

**BTH（Base Transport Header，基础传输头）**：传输层的核心头，包含目的 QP 编号、操作类型（是 Send、RDMA Write 还是 Read）、以及 PSN（包序列号）。交换连接信息时的那些 QPN、PSN 最终就填在这个头里，每个 IB 包都有。

**Payload（负载）**：实际传输的数据，长度可变。

**ICRC（Invariant CRC，不变循环冗余校验）**：覆盖包中端到端传输过程不会改变的字段，做端到端的完整性校验。

**VCRC（Variant CRC，可变循环冗余校验）**：覆盖整个包，每一跳都会重新计算，做链路级的完整性校验。

ICRC 和 VCRC 的分工是个有意思的设计：有些字段（比如转发过程中可能被修改的部分）逐跳会变，有些字段端到端恒定。VCRC 每跳重算保证链路传输没出错，ICRC 端到端不变保证核心数据从头到尾没被破坏。两层校验各管一段。

## 8.8 RDMA 的软件栈：内核驱动、rdma-core 与 MLNX_OFED

做 RDMA 开发的首选平台是 Linux，这一节我们来谈谈 Linux 平台上 RDMA 软件栈是如何分工的。

要理解这套东西，先记住一个最关键的事实：**RDMA 软件栈分内核态和用户态两层**

**内核态：驱动早就在主线内核里了**

真正意义上的 RDMA 驱动是一组内核模块，它们随 Linux 主线内核一起发布，我们装好 Ubuntu 的那一刻就已经有了，不需要单独安装。常见的有：

- `ib_core`：RDMA 子系统的核心框架，所有 RDMA 设备的公共底座。
- `mlx5_core` / `mlx5_ib`：NVIDIA/Mellanox ConnectX 系列网卡的驱动。
- `rdma_rxe`：Soft-RoCE 的内核实现，让没有 RDMA 网卡的普通以太网卡也能用软件模拟跑 RDMA，前面折腾 Soft-RoCE 用的就是它。

这意味着：在一台装好的 Ubuntu 上，RDMA 的内核能力是现成的，缺的只是上层的库和工具。

**用户态：rdma-core 提供库和工具**

`rdma-core` 这个包提供的是**用户态**的部分，也就是应用程序和内核驱动之间的桥梁：

- `libibverbs`：Verbs 库，RDMA 编程调用的 API 就来自这里。
- `librdmacm`：RDMA CM 库，负责连接管理。
- 各网卡的用户态 provider 库（如 `libmlx5`），负责把通用 Verbs 调用对接到具体硬件。
- 一批诊断工具：`ibv_devinfo`、`ibstat`、`rdma`、`ibping` 等，用来观察 LID、GID、端口状态、子网管理器等信息。

在 Ubuntu 上安装 rdma-core：

```bash
apt install rdma-core
```

总结：**主线内核提供 RDMA 驱动（内核态），`rdma-core` 补上 Verbs 库、CM 库和诊断工具（用户态），两者合起来才是一套能用的开源 RDMA 软件栈。** 这套"主线内核 + rdma-core"的组合，正是历史上那个独立大包 OFED 演进后的当代形态：用户态收敛成了 `rdma-core`，内核态并入了主线，发行版开箱即用。

**MLNX_OFED**

既然开源栈开箱即用，那 NVIDIA 的 **MLNX_OFED** 是干什么的？

MLNX_OFED 是 NVIDIA 针对自家网卡打包的**整套替换方案**。它不是在开源栈上打补丁，而是用 NVIDIA 自己编译的一整套组件，把上面提到的开源组件**整体顶替掉**：

- 内核态：用 NVIDIA 自己的 `mlx5` 模块替换主线内核自带的版本。
- 用户态：用 NVIDIA 自己的 `libibverbs`、`libmlx5` 等替换 `rdma-core`。

安装时它会卸载或覆盖系统原有的 `rdma-core`，之后系统里跑的就是 NVIDIA 这一整套，而不是发行版主线那套了。

那什么时候才需要它？大致是这几种情况：

- 用的是 NVIDIA/Mellanox 网卡，且想要官方的性能优化、最新特性，或厂商技术支持。
- 运行在较老的发行版/内核上，需要 NVIDIA 提供的兼容性支持。
- 要用 GPUDirect RDMA 等需要配套驱动栈的高级特性。

反过来，如果只是学习、实验，或者用 Soft-RoCE 跑 lab，**完全不需要 MLNX_OFED，主线内核 + `rdma-core` 就够了**，而且更干净，跟内核版本天然同步，不用操心模块和内核不匹配的问题。

具体的安装步骤和工具用法，前面 Soft-RoCE 那一章已经实践过，这里不再重复。
