# 第十八章：Infiniband Fabric 的在网计算（SHARP）

## 18.1 从 AllReduce 的"搬运成本"说起

前面几章我们一直在围绕一件事打转：怎么让流量在 fabric 里走得更好——更均衡（自适应路由）、更隔离（分区）、更有序（QoS）、不拥塞（CC）。但它们有一个共同的前提：**网络的职责，始终是"把数据从 A 搬到 B"。**

这一章要介绍的 SHARP，则跳出了这个前提。它问的是一个更激进的问题：**网络能不能不只是"搬"数据，而是顺手把数据"算"了？**

这个问题的由来，还是 AI 训练里那个老熟人：**AllReduce**。

回忆一下 AllReduce 在干什么：成千上万块 GPU 各自算出一份梯度，需要把所有人的梯度**求和**，再把这个和分发回每一块 GPU。传统做法（如 ring-allreduce）是让数据在 GPU 之间一段段传递、边传边加。问题在于：

- **同一份梯度数据要在网络里来回穿好多趟**。GPU 越多，总搬运量越大；
- 第十四章的自适应路由(AR)能把这些流量摊匀、避免热点，但它**改变不了"数据被搬运的总量"**，该搬的还得搬。

于是有人想：既然这些梯度最终只是要"加起来"，而它们反正都要路过交换机，那**能不能让交换机在转发的途中就把它们加掉**？这样网络里流动的就不再是一份份原始梯度，而是已经被逐级求和、越来越小的"局部结果"。

这就是 **SHARP（Scalable Hierarchical Aggregation and Reduction Protocol，可扩展分层聚合与归约协议）** 的核心思想，也是业界常说的 **在网计算（In-Network Computing）** 的代表。

---

## 18.2 SHARP：把"归约"搬进交换机

SHARP 做的事情，一句话概括：**把集合通信里的"归约（reduction，比如求和）"操作，从 GPU 卸载到交换机硬件里完成。**

它的载体是一棵 **聚合树（SAT，SHARP Aggregation Tree）**。支持 SHARP 的交换机（NVIDIA Quantum 系列）内部带有一个硬件归约引擎，称为 **聚合节点（AN，Aggregation Node）**；这些 AN 在 fabric 之上组织成一棵逻辑树：

```
                 ┌─────────┐
                 │  Root   │   ← 根：算出最终结果，再沿树往下分发
                 │   AN    │
                 └────┬────┘
            ┌─────────┴─────────┐
       ┌────┴────┐         ┌────┴────┐
       │ Switch  │         │ Switch  │   ← 中间交换机：把孩子的数据求和后上送
       │   AN    │         │   AN    │
       └──┬───┬──┘         └──┬───┬──┘
        ┌─┘   └─┐           ┌─┘   └─┐
       GPU     GPU         GPU     GPU    ← 叶子：把本地梯度送上树
```

一次 AllReduce 在 SHARP 上的过程，变成了"一上一下"两趟：

1. **上行聚合（reduce-up）**：每块 GPU 把本地梯度送给它所在的叶子交换机；每台交换机的 AN 把**它所有孩子的数据求和**，只把这个"局部和"往上一级送；逐级汇聚，直到根算出**全局和**。
2. **下行分发（broadcast-down）**：根把这个全局结果沿树往下逐级广播，最终每块 GPU 都拿到同一个最终结果。

和传统 ring-allreduce 一比，差别是根本性的：

|            | 传统 ring-allreduce      | SHARP 在网归约               |
| ---------- | ------------------------ | ---------------------------- |
| 归约由谁算 | GPU 之间互相传、边传边加 | **交换机里的 AN 硬件算**     |
| 数据搬运量 | 同一数据来回穿网多趟     | **一上一下两趟即可**         |
| 延迟随规模 | 随节点数近似线性增长     | **随树高对数增长**，更可预测 |
| GPU 占用   | 要参与每一步收发与求和   | **从归约计算中解放出来**     |

除了求和，SHARP 还支持 min/max 等多种归约算子，以及 barrier、broadcast 等集合操作。新一代（SHARPv2 起，配合 Quantum-2/NDR）还引入了面向大块数据的流式聚合、浮点支持等，把适用范围从"小消息的延迟敏感归约"扩展到了"大梯度的带宽密集归约"。

> Adaptive Routing 是"把 AllReduce 的流量摊匀"，SHARP 则是"从根上少搬甚至不搬这些流量"。一个优化搬运路径，一个减少搬运本身。在超大规模训练里，两者常常是叠加使用的。

## 18.3 SHARP 的组件：谁来建树、谁来算

和 CC 类似，SHARP 也不是 OpenSM 一个人能包办的，它有自己的一套软硬件组件：

- **聚合节点（AN）**：交换机里的硬件归约引擎，真正"做加法"的地方，依赖 NVIDIA Quantum 系列交换机。
- **聚合管理器（AM，Aggregation Manager）**：一个独立的软件实体（通常随 SHARP 软件栈或 UFM 部署）。它负责**发现支持 SHARP 的交换机、构建并管理聚合树、分配树资源**。它与 OpenSM 并存，但职责独立，**OpenSM 管路由与子网，AM 管聚合树**。
- **SHARP 守护进程（sharpd）**：跑在计算节点上，作为应用与 SHARP fabric 之间的桥梁。
- **应用接入**：NCCL 通过 CollNet 路径调用 SHARP，把 AllReduce 卸载到网络；MPI 也可借 SHARP 加速其集合操作。

那 OpenSM 在这里负责什么？**仅仅是"在支持的交换机上把 SHARP 这个能力打开"**，真正的建树和运行，交给 AM。

## 18.4 OpenSM 的 SHARP 配置选项

在 OpenSM 配置文件里，和 SHARP 相关的就一个开关：

```bash
# SHArP support
# 0: Ignore SHArP - No SHArP support
# 1: Disable SHArP - Disable SHArP on all supporting switches
# 2: Enable SHArP - Enable SHArP on all supporting switches
sharp_enabled 2
```

有几个注意点值得强调：

1. **OpenSM 在 SHARP 这件事上只是个"总闸"**。它做的只是"在硬件支持的交换机上把能力使能/禁用"，至于聚合树怎么建、归约怎么跑，全在 AM 和交换机硬件那边。
2. **注意那句反复出现的"on all supporting switches"**。它只对**声明了 SHARP 能力的交换机**生效。

## 18.5 小结

**SHARP 把网络从"搬运者"变成了"参与者"。** 它的核心，是把 AI 训练里最沉重的集合通信（AllReduce 的归约（求和）操作）从 GPU 卸载到交换机硬件中完成。

**机制是一棵聚合树上的"一上一下"**：叶子 GPU 把梯度送上树，每台交换机的聚合节点（AN）把孩子们的数据逐级求和、只上送局部结果，根算出全局和后再沿树下发。结果是：数据搬运量大幅下降、延迟随树高对数增长、GPU 从归约计算中解放出来。

**组件上各司其职**：AN 是交换机里做加法的硬件引擎，AM（聚合管理器）负责建树与管理，sharpd 把应用接进来，而 OpenSM 只用一个 `sharp_enabled` 开关在支持的交换机上把能力打开。

**硬件特性**：依赖 NVIDIA Quantum 交换机与 SHARP 软件栈，ibsim 够不着，所以本章以概念为主。

从Partition、QoS、CC 到 SHARP，IB 一步步把越来越多的"智能"沉到了网络本身。而 SHARP 走得最远，**它让网络不再只是计算的通道，而成了计算的一部分。** 这正是 InfiniBand 在大模型训练场景中难以被轻易替代的关键原因之一。"在网计算"这个方向，也是以太网阵营仍在努力追赶的目标。
